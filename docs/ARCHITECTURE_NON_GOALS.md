# Poe Orchestrator — Architecture Non-Goals

What Poe deliberately does NOT do, and why. Use this to say no cleanly.

---

## Tool minimalism

**Not doing:** Reducing to a 4-tool set (read/write/edit/bash) like Pi coding agent.

**Why not:** Poe is a multi-day autonomous orchestrator, not an interactive coding assistant. The tools it has beyond basic I/O are load-bearing:
- `schedule_run` — deferred execution across sessions
- `create_team_worker` — multi-agent delegation
- `flag_stuck` — explicit stuck signal enabling the recovery loop

Minimalism is a virtue, but Pi's 4-tool constraint fits a coding REPL. Poe's tool set fits a mission executor. They're different use cases.

---

## MCP as primary integration path

**Not doing:** Treating MCP as the default way to add tools (Phase 41 step 7 is additive, not a replacement).

**Why not:** MCP servers are useful for external system access but add a process-boundary dependency. Most tool extensions are better served by Phase 41's in-process tool registry. MCP integration is available for cases that require it; it's not the default pattern.

---

## Interactive / approval-gated execution

**Not doing:** Permission popups, plan-first mode, or approval gates before each step.

**Why not:** Poe runs over hours and days. User-in-the-loop interrupts break the autonomous execution model. Safety is handled through:
- Constraint enforcement at compose-time (constraint.py)
- Inspector + quality gate for output quality
- Pre-step event bus for programmatic veto (step_events.py)
- Explicit `flag_stuck` when Poe genuinely needs human input

The user configures guard rails upfront; they shouldn't be the guard rail.

---

## Sub-agent spawning without visibility

**Not doing:** Hidden sub-agent calls that don't surface in session state or the inspector.

**Why not:** This was one of the explicit pain points in Mario Zechner's Pi analysis, and it's why Poe has the inspector panel and thinkback replay. Every team worker spawn is visible in the loop context and logged to outcomes. No black boxes.

---

## Full Neo4j / graph database for memory

**Not doing:** A full property graph backend for memory, lesson, and failure storage.

**Why not:** JSONL flat files + SQLite for failure adjacency covers all current use cases. Failure nodes with `caused_by` edges (the Mimir pattern worth stealing) can be SQLite adjacency tables. Neo4j is overkill until there are thousands of failure nodes with complex multi-hop queries. The threshold for this upgrade is "JSONL + SQLite causes real query performance problems," not "it would be interesting."

---

## Plugin/extension marketplace

**Not doing:** Downloadable plugin ecosystem, extension registry, or plugin installation UI.

**Why not:** Extensions are generated by Poe, not downloaded. The self-extending pattern (Pi's key insight) is the right model: ask the agent to add the capability. This keeps the system prompt footprint small and avoids dependency sprawl. Tool definitions live in `tool_registry.py` and can be generated at runtime via Phase 41.

---

## Language-model-agnostic portability guarantees

**Not doing:** Full session portability across LLM providers as a first-class contract.

**Why not:** `llm.py` already handles multi-provider routing (Anthropic, OpenRouter, OpenAI, subprocess). Guaranteeing that a session started on Claude Opus can seamlessly continue on Gemini requires additional serialization work with marginal real-world benefit. Provider switching is supported for individual calls; cross-provider session continuity is not a design goal.

---

## Headless UI framework

**Not doing:** A production-grade terminal UI, TUI library, or flicker-free rendering.

**Why not:** Poe is headless. Primary interfaces are Telegram and Slack. The dashboard (Phase 23) is an internal observability tool, not a product UI. Time spent on UI polish is time not spent on autonomy and reliability.

---

## Self-hosted model primary support

**Not doing:** First-class support for self-hosted models (Ollama, vLLM, etc.) as primary execution targets.

**Why not:** Self-hosted models don't reliably support tool calling at the quality level Poe requires. The architecture supports them via `subprocess` adapter for specific use cases (eval runs, fast drafts), but they're not a primary execution path. Frontier model quality is load-bearing for the recovery loop, clarification milestone, and quality gate.

---

## When to revisit this list

- Revisit MCP-as-default if external tool server ecosystem matures and per-call latency drops below ~200ms
- Revisit Neo4j if failure node count exceeds ~10k and multi-hop queries become frequent
- Revisit provider portability if Jeremy decides to run primary sessions on a non-Anthropic model
