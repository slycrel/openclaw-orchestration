# Burn-in trial — OpenClaw substrate path (2026-07-02)

The P4 trial from the substrate arc ("get the project where we can trial it
for real with hermes or openclaw"). 14 goals dispatched through the installed
adapter (`~/.openclaw/workspace/scripts/maro-dispatch.sh` → enqueue → drain →
run → run_card → Telegram), plus 4 organic runs in the same window. Each batch
was adjudicated by hand against the artifacts on disk before the next was sent.

## Verdict

**The pipeline works.** Every dispatched goal flowed dispatch → run →
project-dir artifacts → run card → Telegram DM with no manual intervention.
The *work* side was healthy from the start: 12 of 14 dispatched goals
delivered correct artifacts (the other 2: one deliberately-impossible control,
one rate-limit casualty). What the burn-in actually caught was the **judging**
layer — four verdict-integrity bugs and a routing bug, all found by comparing
recorded verdicts against artifacts on disk, all fixed and re-proven live
within the trial.

## What the burn-in caught (all fixed same-day)

1. **Closure cwd false negatives** (batch 1: 3/3 wrong verdicts). Closure
   checks ran in Maro's launch cwd, not the project dir — every artifact probe
   failed on work that was fully delivered. `verify_goal_completion` now
   backfills empty `workspace_path` from the run-scoped subprocess cwd
   (ec4c1f3). Pre-fix `goal_achieved` data for non-repo goals is
   false-negative-poisoned.
2. **Inconclusive-as-failure false negatives** (batch 2: 2 wrong verdicts).
   Any single inconclusive probe (often the verifier's own malformed command)
   mechanically flipped `complete=True→False`, and the verdict prompt pushed
   the LLM the same direction ("Goal achieved." at 0.95 confidence recorded as
   not achieved). Positive-evidence rule now: passing checks are proof; the
   flip fires only when zero checks passed (9be749b).
3. **NOW-lane misrouting** (batch 3). "Summarize X … saved to artifacts/…"
   routed NOW off the leading verb; NOW answers inline and cannot write files,
   so the self-verdict honestly demoted a run that never had a chance.
   Capability override in `intent.classify`: goals naming a file deliverable
   force agenda (8ed0a09). Re-proven live in batch 4 (same shape → agenda).
4. **Skipped-closure false positive** (batch 4). A rate-limit-stuck run
   recorded `goal_achieved=True` because the fail-open null verdict
   ("Verification skipped.", checks_run=0) was written unconditionally.
   Verdicts are recorded only when checks actually ran (90b4d1b).
5. **Cost-per-goal was unrecoverable** — no join key between runs and
   step-costs. `loop_ids` now recorded in run metadata; run cards carry
   `total_cost_usd` (2989bb0). Also: status `incomplete` classed `unknown` →
   `partial`; closure events now log gap text.

## Environmental finding (not a Maro bug)

The box's `backend_order` rides the `claude -p` subprocess for MODEL_POWER —
the same subscription interactive sessions use. Batch 4 hit the shared rate
limit; failover found OpenRouter with no credit (402) and a stale OpenAI
config (404). Degradation was graceful (clean backoff bails, navigator
escalate-to-human at dispatch).

**Resolved 2026-07-02 (Jeremy): accept the contention** — the subscription
stays the shared lane. Rate-limit stucks are an accepted operating cost; the
graceful-degradation path is the designed behavior.
